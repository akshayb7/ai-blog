---

**TL;DR:**
- Quantization reduces weight precision from 16-bit to fewer bits — 4-bit uses 25% of original memory, but quality falls off a cliff on large models
- The degradation is non-linear: q4 on a 20B model looks bad; q6 looks good; the jump between them is dramatic
- mflux always downloads full bf16 weights and quantizes at load time — the download size doesn't change regardless of quantization level
- Distilled models (Klein) can't use negative prompts or custom guidance because they eliminated the mechanism that makes those features work
- Classifier-Free Guidance requires running inference twice; distillation trades that away for speed

## What Quantization Actually Does to Model Weights

A neural network is a massive collection of numbers — **weights** — each stored at some level of precision. The default is [bfloat16](https://en.wikipedia.org/wiki/Bfloat16_floating-point_format): 16 bits per weight. A 20B parameter model at bf16 is roughly 40GB of raw weights.

Quantization reduces that precision. If a weight is `0.73291`:

- At 8-bit, it might store as `0.73` — close, rounding error is small
- At 6-bit, it stores as `0.72` — rounding error grows
- At 4-bit, it stores as `0.7` — noticeably imprecise

Each individual rounding error is tiny. Across 20 billion weights, those errors compound — especially in the layers responsible for fine-grained detail.

The memory math is roughly linear:

| Quantization | Bits per weight | Memory vs bf16 |
|-------------|-----------------|----------------|
| bf16 (full) | 16 | 100% (~40GB for Qwen) |
| q8 | 8 | ~50% (~25GB) |
| q6 | 6 | ~37.5% (~20GB) |
| q4 | 4 | ~25% (~14GB) |

The memory reduction is real. The quality impact is not proportional.

## mflux Always Downloads the Full Model

I expected this to work like downloading a pre-quantized checkpoint — pick `-q 6`, get a 20GB download.

That's not what happens. mflux downloads the full bf16 weights regardless of quantization level, then quantizes them in memory at load time. Qwen is always a ~54GB download (including all the model components). The quantization happens at runtime, the first time the model is loaded.

The practical consequence: even at q4, the initial load briefly requires enough memory to hold the full weights. On a 48GB machine, loading Qwen at q8 pushed the system close to the ceiling — the process appeared to die silently before eventually completing after an extended load time.

The design makes sense from mflux's perspective: one canonical checkpoint, flexible quantization at runtime. From my perspective, it meant ~54GB of disk space regardless of how I planned to run it.

There is a workaround for subsequent runs: `mflux-save` lets you persist the quantized weights to disk after the first load. Run it once with `--quantize 6` and save to a local path — future loads skip the full-model quantization step and load the smaller saved file directly. I didn't discover this until after running several experiments.

```bash
# Check your model name with: mflux-save --help
mflux-save \
  --path "/Users/me/models/qwen-q6" \
  --model qwen-image \
  --quantize 6
```

## The Cliff: q4 vs q6 vs q8 on Qwen

Same prompt, same seed (`7742`), same resolution (`1280×864`), same steps (`30`). Only the quantization level changed.

```bash
mflux-generate-qwen \
  --prompt "The Colosseum in Rome at golden hour, with a vintage red Vespa scooter parked in the foreground, Italian cypress trees, cobblestone street" \
  --steps 30 -q 4 --seed 7742 --width 1280 --height 864
```

**q4 — 17.5 minutes, ~14GB RAM:**

![Qwen at q4: painterly artifacts, degraded textures, oil-paint quality degradation](/images/qwen-rome-q4.png)

*Stone detail lost. Sky looks painted. The Vespa barely reads as a vehicle.*

**q6 — 18.3 minutes, ~20GB RAM:**

![Qwen at q6: clean photorealistic Colosseum with Vespa](/images/qwen-rome-q6.png)

*Dramatic jump. Stone textures visible, sky gradient clean, Vespa recognizable. Same prompt, 0.8 minutes longer.*

**q8 — 21.4 minutes, ~25GB RAM:**

![Qwen at q8: sharpest result, highest detail](/images/qwen-rome-q8.png)

*Noticeably sharper than q6 — edge definition, texture depth. Best output, but the load nearly saturated available RAM.*

The q4 result isn't an artistic choice. It's degradation. What surprised me was the gap between q4 and q6 — not a gradual slide, but a cliff.

## Why the Degradation Isn't Linear

This is the part that took me a while to internalize.

Diffusion models generate images by iteratively refining from noise. Each denoising step has the model predict what should be removed — where noise ends and signal begins. That prediction requires the model to maintain accurate internal representations of textures, edges, gradients.

Those representations live in the weights. At 4-bit, you're discarding roughly 75% of the precision those representations were trained with. A single miscalculated weight might do nothing visible. But error accumulates across layers — each layer's output becomes the next layer's input. The compounding doesn't happen linearly.

Fine-grained detail is the first casualty. Stone texture requires the model to hold subtle weight patterns simultaneously across many neurons. Sky gradients require smooth transitions between similar values. Both depend on precision that 4-bit simply doesn't have enough bits to encode.

At q6, enough precision survives that the compounding error stays bounded. At q4, it doesn't. That's the cliff.

It's also model-size dependent. Klein at 4B runs at full bf16 precision and uses ~8GB — no quantization needed at all. Dev at 12B runs cleanly at q8, ~13GB, with minimal visible impact. The larger the model, the more precision matters for the weights holding complex detail. A 20B model at q4 is more damaged than a 12B model at q4 would be, because you're compressing more parameters with the same bit depth.

**Rule of thumb:** use the largest model that fits comfortably in memory at the highest quantization level that leaves headroom for the OS. On 48GB: Klein at full precision, Dev at q8, Qwen at q6.

## Distillation: What Klein Gave Up for Speed

Klein generates an image in 12 seconds. Dev takes 1.5 minutes minimum. That gap has a specific cause.

**Knowledge distillation** is a training technique where a smaller "student" model is trained to imitate a larger "teacher" model. Klein (4B parameters) was step-distilled from the **FLUX.2 9B base model** — a separate, newer architecture from FLUX.1. The student learned to converge in 4 steps what the teacher needed many more to produce.

The payoff: Klein only needs **4 inference steps**. Dev needs 20. The speed difference is almost entirely step count.

But distillation also changed the inference architecture in ways that matter for how you use the model.

## Why Distilled Models Can't Use Negative Prompts

The error I hit immediately:

```
Error: negative-prompt is not supported for FLUX.2
```

This looks like a missing feature. It's actually an architectural consequence.

**Classifier-Free Guidance (CFG)** is how base models process negative prompts. The mechanism requires running inference twice per step:

1. Once **with** the prompt (or positive conditioning)
2. Once **without** (or with the negative prompt)

The final output is interpolated between the two: steer toward the positive, steer away from the negative. The guidance scale (`--guidance`) controls how aggressively to interpolate. At 3.5, you get balanced adherence. At 7.0, the model strictly follows the prompt at some cost to naturalness.

This dual-pass is expensive — it roughly doubles inference cost per step. But it's what gives you steering control.

Distilled models eliminate the dual-pass. Klein was trained to produce good results in a single forward pass, with the "avoid bad outputs" behavior baked into the weights during distillation training. There's no separate conditioning pathway for negative prompts. The architecture doesn't have a place to plug them in.

Same reason guidance is fixed:

```
Error: guidance is only supported for FLUX.2 base models. Use --guidance 1.0
```

The guidance scale controls CFG interpolation. No CFG mechanism, no guidance scale knob. Klein runs at guidance 1.0 because 1.0 means "don't interpolate" — a single-pass result. The guidance behavior is "compiled in" during distillation training.

These aren't bugs that future mflux versions will fix. They're consequences of the architecture Klein was trained with.

## Inference Steps and Guidance: What the Parameters Actually Control

**Steps** (`--steps`): Image generation starts from pure random noise and iteratively removes it. Each step refines the output — more noise removed, more structure added. More steps means more refinement passes.

Distilled models converge in 4 steps because they were trained to. Base models need 20-50 because each step is doing less work and quality accumulates more gradually. Going beyond the sweet spot has diminishing returns — Dev at 40 steps versus 20 is marginal; Dev at 4 steps looks unfinished.

**Guidance scale** (`--guidance`): Controls how strictly the model follows the text prompt during CFG interpolation.

| Range | Behavior |
|-------|----------|
| 1–2 | Loose — creative interpretation |
| 3–4 | Balanced — default (3.5) |
| 5–7 | Strict — heavy prompt adherence |
| 8+ | Degrades — over-optimization artifacts |

At very high guidance, the model over-optimizes for the prompt at the expense of visual coherence. It's like asking someone to literally follow instructions until they stop making sense.

## Why Unified Memory Lets This Run Locally At All

Quick version, since [the previous post](/posts/running-image-gen-locally-apple-silicon) covers this in more depth:

Traditional NVIDIA GPUs have a separate VRAM pool — 8GB, 16GB, 24GB. The GPU cannot directly access system RAM. Running a 20B model requires a GPU with 20GB+ VRAM; Qwen at q6 (20GB) would require an A100 or similar.

Apple Silicon uses **unified memory** — CPU and GPU share the same physical RAM pool. On a 48GB Mac Mini, the GPU has access to all 48GB. That's why Qwen at q6 fits locally on hardware that cost $1,200.

Quantization is still necessary — not to fit GPU VRAM, but to fit within the unified pool while leaving headroom for the OS and other processes. Without quantization, Qwen's 40GB weights would consume nearly all 48GB, leaving nothing for the runtime.

## Practical Recommendations

| Model | Size | What to run | RAM at that setting |
|-------|------|-------------|---------------------|
| Klein 4B | 4B | Full precision (no `-q`) | ~8GB |
| FLUX.1 Dev | 12B | `-q 8` | ~13GB |
| Qwen Image | 20B | `-q 6` on 48GB; `-q 4` if ≤32GB | ~20GB / ~14GB |

For machines with 24GB or less unified memory, Qwen at any quality level is borderline. Klein and Dev at q8 are the practical options.

Don't reach for q4 on a 20B model and expect usable output. The memory savings aren't worth the quality cliff unless you're doing structural exploration — checking composition, rough color palette — and plan to regenerate at a higher quantization level.

## What I Still Haven't Tested

Head-to-head: Dev at q8 vs Qwen at q6, identical prompt, controlled for resolution. My hypothesis is Dev wins on generic photorealism; Qwen wins where world-knowledge accuracy matters (real buildings, recognizable objects). That's not a data-backed conclusion yet.

Resolution scaling on Dev is also open. The benchmarks I have are at 512×512 and 768×1024. How quality degrades as you push toward 1280×864 with 20 steps is something I want to measure before making strong claims.
